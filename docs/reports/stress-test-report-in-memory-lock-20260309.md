# 스트레스 테스트 리포트 — 비관적 락 vs 인메모리 락 비교

> 테스트일: 2026-03-09
> 환경: EC2 t3.small (2vCPU, 2GB)
> 관련: 비관적 락 → 인메모리 락 전환

---

## 1. 테스트 목적

비관적 락(`SELECT FOR UPDATE`)에서 인메모리 락(`ReentrantLock`)으로 전환한 후,
**DBCP 커넥션 풀 고갈 문제가 해소되었는지** 검증한다.

## 5. 분석

### 5-1. 핵심 성과: DBCP 커넥션 풀 고갈 해소

비관적 락의 근본 문제는 **락 대기가 `@Transactional` 안에서 발생**하여 DB 커넥션을 점유하는 것이었다.

```
[비관적 락 — 500 VU]
Thread-1:    커넥션 점유 → SELECT FOR UPDATE → 락 획득 → 로직 → COMMIT
Thread-2~30: 커넥션 점유 → SELECT FOR UPDATE → 락 대기 ⏳ (커넥션 낭비)
Thread-31~200: 커넥션 대기 ⏳ (pending: 169)

→ 30개 커넥션 중 29개가 락 대기로 낭비
→ 170개 스레드가 커넥션조차 못 받음
```

인메모리 락은 **락 대기를 DB 바깥(JVM 메모리)으로 분리**하여 이 문제를 해소했다.

```
[인메모리 락 — 500 VU]
Thread-1:     락 획득 → @Transactional → 커넥션 획득 → 로직 → COMMIT → 커넥션 반환
Thread-2~500: JVM 메모리에서 락 대기 ⏳ (커넥션 사용 안 함)

→ 활성 커넥션 1개, pending 0개
→ 나머지 29개 커넥션은 다른 API가 자유롭게 사용 가능
```

### 5-2. 처리량 감소 (57 → 45 req/s)의 원인

비관적 락이 처리량이 더 높은 이유는 **DB 엔진의 락 큐잉이 JVM의 락 큐잉보다 병렬성이 높기 때문**이다.

- 비관적 락: 30개 스레드가 동시에 DB 커넥션을 잡고 대기 → 락 해제 시 즉시 다음 스레드가 실행
- 인메모리 락: 1개 스레드만 통과 → 나머지는 JVM에서 순차 대기 → 락 해제 → 다음 스레드가 커넥션 획득부터 시작

그러나 이 처리량 차이는 **커넥션 풀 고갈이라는 대가를 치른 결과**이다. 비관적 락은 30개 커넥션과 200개 스레드를 전부 소진하면서 57 req/s를 달성했고, 인메모리 락은 커넥션 1개만 사용하면서 45 req/s를 달성했다.

### 5-3. 응답 시간이 여전히 느린 이유

단일 스케줄에 500 VU가 집중되는 테스트 특성상, **한 번에 1개 요청만 처리**하는 직렬화 병목은 락 방식과 무관하게 존재한다.

응답 시간의 대부분은 비즈니스 로직이 아니라 **락 대기 시간**이다. 이것은 단일 자원에 500명이 동시에 접근하는 구조적 한계이지, 락 구현의 문제가 아니다.

### 5-4. 리소스 효율성 비교

| 관점 | 비관적 락 | 인메모리 락 |
|------|----------|------------|
| DB 커넥션 사용 | 30개 (전부 소진) | 1개 |
| 다른 API 영향 | 커넥션 고갈로 전체 API 영향 | 예약 외 API 정상 동작 |
| CPU | 75% | 60% |
| 스케일 여유 | 없음 (한계 도달) | 있음 (커넥션 29개 여유) |

## 8. 결론

인메모리 락 전환의 목적은 **처리량 향상이 아니라, DBCP 커넥션 풀 고갈 해소**였다.

| 목표 | 결과 |
|------|------|
| DBCP pending 해소 | 169개 → **0개** |
| DBCP active 감소 | 30/30 → **1/30** |
| CPU 사용량 감소 | 75% → **60%** |
| 서버 에러율 유지 | 0.00% → **0.00%** |

처리량은 57 → 45 req/s로 감소했지만, 이는 비관적 락이 **시스템 리소스를 전부 소진하면서 달성한 수치**이다. 인메모리 락은 커넥션 1개만 사용하면서도 안정적으로 동작하며, 나머지 리소스는 다른 API가 사용할 수 있다.
