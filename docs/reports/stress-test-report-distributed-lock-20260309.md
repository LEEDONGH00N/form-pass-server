# 스트레스 테스트 리포트 — 비관적 락 vs 분산 락 동일 인프라 비교

> 테스트일: 2026-03-09
> 환경: EC2 t3.small (2vCPU, 2GB) + Redis (같은 인스턴스)
> 관련: ADR-003

---

## 1. 테스트 목적

동일한 인프라(t3.small)에서 비관적 락(`SELECT FOR UPDATE`)과 Redisson 분산 락의
**성능 지표와 시스템 리소스 사용량을 비교**하여, 분산 락 도입의 근거를 검증한다.

## 4. 분석

### 4-1. 에러율: 둘 다 0.00%

500 VU 환경에서 비관적 락과 분산 락 모두 서버 에러 없이 처리되었다.
t3.small(2vCPU, 2GB) 환경에서는 비관적 락도 500 VU를 감당할 수 있다.

### 4-2. 처리량: 비관적 락 57 req/s vs 분산 락 44 req/s

비관적 락의 처리량이 29% 더 높다.
분산 락은 Redis를 경유하는 오버헤드(네트워크 I/O, Pub/Sub 대기)가 있기 때문이다.
단일 인스턴스 환경에서 순수 처리량만 보면 비관적 락이 유리하다.

### 4-3. 비관적 락의 구조적 위험 — DBCP pending 169개

에러율은 0%였으나, 모니터링 지표는 **시스템이 한계에 도달한 상태**를 보여준다.

```
[비관적 락 — 500 VU 시점의 시스템 상태]

톰캣 스레드:  200/200 busy (유휴 스레드 0개)
DB 커넥션:   30/30 active (전부 사용)
DB pending:  169개 (커넥션을 기다리는 스레드)
CPU:         76% (여유 거의 없음)
```

**비관적 락은 락 대기 중에도 DB 커넥션을 점유하거나, 커넥션 풀에서 대기하는 구조**이다.
현재 500 VU에서는 버텼지만, 다음 상황에서 커넥션 대기 타임아웃 → 5xx 에러가 발생할 수 있다:

- VU가 더 높아지는 경우 (700+)
- 인스턴스 리소스가 줄어드는 경우
- 트랜잭션 처리 시간이 길어지는 경우 (복잡한 비즈니스 로직 추가 등)

### 4-4. 분산 락의 구조적 이점

분산 락은 락 대기를 Redis가 담당하므로, DB 커넥션은 **실제 트랜잭션 처리 시간만** 점유한다.

```
[비관적 락]
요청 → DB 커넥션 획득 → SELECT FOR UPDATE (락 대기 + 처리) → 커넥션 반환
        └────────────── 전체 시간 DB 커넥션 점유 ──────────────┘

[분산 락]
요청 → Redis 락 대기 (DB 커넥션 없음) → 락 획득 → DB 커넥션 획득 → 처리 → 커넥션 반환
                                                   └── 최소 시간만 점유 ──┘
```

이로 인해:
- DB 커넥션 풀 고갈 위험이 구조적으로 낮음
- VU가 높아져도 DB 커넥션 pending이 급증하지 않음
- 시스템의 확장 여유(headroom)가 더 큼

### 4-5. 분산 락의 추가적 이점

성능 외에도, 비관적 락은 다음 상황에서 **구조적으로 동작하지 않는다**:

| 시나리오 | 비관적 락 | 분산 락 |
|----------|----------|--------|
| 스케일아웃 (인스턴스 2대+) | 인스턴스 간 락 공유 불가 | Redis가 중앙 락 역할 |
| 무중단 배포 (Blue/Green) | 구 버전·신 버전 간 락 미공유 | 동일 Redis로 공유 |

현재는 단일 인스턴스이지만, 향후 스케일아웃이나 무중단 배포를 도입하면
비관적 락은 즉시 무효화되고, 그 시점에 분산 락으로 전환하는 마이그레이션 비용이 발생한다.

## 7. 결론

500 VU 환경에서 비관적 락과 분산 락 모두 서버 에러 없이 동작했다.
순수 처리량만 보면 비관적 락(57 req/s)이 분산 락(44 req/s)보다 높다.

그러나 **비관적 락은 시스템이 한계에 도달한 상태에서 동작**하고 있었다:
- DB 커넥션 풀 전부 가동, pending 169개
- 톰캣 스레드 200개 전부 busy
- CPU 76%

분산 락을 도입한 이유는 단순한 성능 개선이 아니라, **구조적 안정성과 확장성** 확보이다:

| 관점 | 비관적 락 | 분산 락 |
|------|----------|--------|
| 단일 인스턴스 500 VU | 동작함 (한계 근접) | 동작함 (여유 있음) |
| 더 높은 부하 | 커넥션 풀 고갈 위험 | Redis가 부하 흡수 |
| 스케일아웃 | 불가 | 가능 |
| 무중단 배포 | 정합성 위험 | 안전 |
