# 스트레스 테스트 리포트 — 분산 락 도입 Before/After 비교

> 테스트일: 2026-03-09
> 환경: EC2 t3.small (2vCPU, 2GB) + Redis (같은 인스턴스)
> 관련: ADR-003

---

## 1. 테스트 목적

비관적 락(`SELECT FOR UPDATE`)을 Redisson 분산 락으로 전환한 후,
500 VU 스트레스 테스트에서 **서버 에러율과 응답시간이 개선되었는지** 검증한다.

## 4. 분석

### 4-1. 서버 에러율: 19.09% → 0.00%

**가장 중요한 개선.** 비관적 락에서 발생하던 5xx 에러가 완전히 사라졌다.

원인:
- 비관적 락: 락 대기 중 DB 커넥션 점유 → 커넥션 풀 고갈 → 타임아웃 → 5xx
- 분산 락: 락 대기를 Redis가 담당 → DB 커넥션은 트랜잭션 처리 시간만 점유

```
[Before — 비관적 락]
500 VU 동시 요청 → 30개 커넥션 점유 + 470개 대기 → 커넥션 풀 고갈 → 5xx

[After — 분산 락]
500 VU 동시 요청 → Redis에서 순차 대기 → 락 획득한 요청만 DB 커넥션 사용
→ 커넥션 풀 고갈 없음 → 5xx 0건
```

### 4-2. 예약 성공: 4,970건 → 6,509건 (+30.9%)

5xx로 실패하던 요청이 정상 처리되면서 성공 건수가 증가했다.

### 4-3. 정상 거부 209건 (3.11%)

429(락 대기 초과)와 409(중복 예약/정원 초과)를 합산한 수치.
시스템이 과부하 상황에서 **에러 대신 명확한 거부 응답**을 반환하는 것이므로 정상 동작이다.

### 4-4. p95 응답시간: 8.21s → 10.65s

p95가 다소 증가했으나, 이는 다음 요인에 의한 것:

1. **단일 스케줄 집중**: 500 VU가 같은 스케줄 1개에 몰리면 락이 순차 처리됨
2. **Before에서는 5xx로 빠르게 실패**: 커넥션 풀 고갈 시 타임아웃으로 즉시 실패 → 응답시간 통계에서 제외
3. **After에서는 모든 요청이 처리됨**: 락 대기 후 성공까지 가므로 전체 응답시간이 길어짐

즉, **Before의 낮은 p95는 "빠르게 실패한 것"이지 "빠르게 처리한 것"이 아니다.**
After에서 p95가 높아진 것은 **이전에 실패하던 요청까지 성공적으로 처리**하기 때문이다.

### 4-5. 실제 운영 환경에서의 예상 성능

이 테스트는 **최악의 시나리오**(500명이 동일한 스케줄 1개에 동시 예약)를 재현한 것이다.
실제 운영에서는:

- 여러 이벤트, 여러 스케줄에 트래픽이 분산됨
- 스케줄별로 별도의 락 키(`schedule:{id}`)를 사용하므로 락 경합이 줄어듦
- p95 응답시간은 이보다 크게 낮아질 것으로 예상

## 7. 결론

| 목표 | 결과 |
|------|------|
| 서버 에러율 해소 | **달성** — 19.09% → 0.00% |
| 초과 예약 방지 | **달성** — 정합성 유지 |
| 처리량 유지/개선 | **달성** — 36 → 44 req/s (+22.6%) |

분산 락 도입으로 **DB 커넥션 풀 고갈 문제가 근본적으로 해결**되었다.
500 VU 환경에서 서버 에러 없이 안정적으로 예약을 처리할 수 있게 되었다.
