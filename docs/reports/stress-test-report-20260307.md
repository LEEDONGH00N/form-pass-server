## 테스트 시나리오

티켓팅 오픈 직후 트래픽이 급증하는 상황을 재현. VU를 단계적으로 올리며 비관적 락(PESSIMISTIC_WRITE)의 병목 지점을 탐색.

```
100 VU (35s) → 200 VU (35s) → 300 VU (35s) → 500 VU (35s)
```

## 성공/실패 판정 기준

| HTTP 상태 | 판정 | 이유 |
|---|---|---|
| 201 | 정상 | 예약 성공 |
| 409, 400 | 정상 | 정원 초과/중복 예약 — 시스템이 올바르게 방어 |
| 5xx | **실패** | 서버 에러/타임아웃 — 비관적 락 병목 |

## 원인 분석

### 병목 원인 (3가지 복합)

| 원인 | 증거 | 에러 유형 |
|---|---|---|
| **DB 커넥션 풀 고갈** | 커넥션 대기 154개 (풀 사이즈 10) | 500 Internal Server Error |
| **t2.micro CPU 크레딧 소진** | CPU 크레딧 기반 인스턴스, 지속 부하 시 성능 급락 | `dial: i/o timeout` |
| **Nginx 업스트림 한계** | Nginx가 500 반환 | `500 Internal Server Error (nginx)` |

### 예약 처리 흐름과 커넥션 점유

```
1. SELECT ... FOR UPDATE  ← 비관적 락 획득 (커넥션 점유 시작)
2. 중복 예약 검증
3. INSERT reservation
4. UPDATE reserved_count
5. COMMIT                 ← 커넥션 반환
```

- HikariCP 기본 커넥션 풀 사이즈: **10개**
- 비관적 락은 트랜잭션이 끝날 때까지 DB 커넥션을 점유
- 500 VU 동시 요청 시 10개 커넥션을 두고 490개 요청이 대기
- 대기 시간 초과 → 5xx 에러 발생

### 에러 로그 패턴

```
# 1. TCP 연결 자체 실패 — 서버가 더 이상 연결을 받지 못함
"dial: i/o timeout"

# 2. 연결 후 응답 없이 끊김 — Spring Boot가 처리 불가 상태
"EOF"

# 3. Nginx까지 도달했지만 업스트림(Spring Boot) 응답 실패
"500 Internal Server Error (nginx/1.24.0)"
```

### 병목 발생 구간

- 100~200 VU: 안정적 처리 (기존 테스트에서 확인된 안전 구간)
- 300 VU~: 응답 시간 급증, 5xx 에러 발생 시작 추정
- 500 VU: p(95) 11.3초, 에러율 13.82%로 한계 초과

## 2차 테스트: 커넥션 풀 조정 (10 → 30)

### 분석

커넥션 풀을 3배로 늘렸음에도 에러율이 오히려 상승했다. 이는 **커넥션 풀 고갈이 병목의 주원인이 아님**을 의미한다.

커넥션을 30개로 늘린 결과:
- 30개 커넥션이 동시에 같은 행(EventSchedule)에 `SELECT ... FOR UPDATE` → **DB 레벨 락 경쟁 심화**
- t2.micro 메모리 1GB에서 커넥션 30개 유지 부담 → 리소스 압박 증가
- p(95)는 개선되었지만, 전체 처리량(RPS)은 오히려 감소 (60.6 → 55.5 req/s)

**결론: 병목의 근본 원인은 비관적 락(단일 행 경합) + t2.micro 인스턴스 한계이다.**
커넥션 풀 튜닝으로는 해결할 수 없으며, 락 전략 자체를 변경해야 한다.

| 지표 | 피크 값 | 한계 | 판정 |
|---|---|---|---|

| HikariCP 대기(Pending) | **169개** | - | 병목 |

### 분석

- **CPU 96% 포화**: t2.micro(vCPU 1)가 한계에 도달. 1차 테스트(0.4)와 비교하면 누적 부하로 CPU 크레딧이 소진된 것으로 보임
- **톰캣 스레드 200개 전부 사용**: 모든 스레드가 요청 처리 중 → 새 요청은 큐에 대기하거나 거부
- **커넥션 풀 30개 전부 사용 + 대기 169개**: 비관적 락 대기로 커넥션을 반환하지 못해 풀 고갈
- 에러율이 1차(13.82%) → 2차(15.96%) → 3차(19.09%)로 **테스트마다 악화**
- 커넥션 풀 튜닝, 타임아웃 설정 등 **애플리케이션 레벨 조정으로는 개선 불가** 확인
- CPU, 스레드, 커넥션이 모두 포화 상태로, **비관적 락 단일 행 경합 + t2.micro 인스턴스 한계**가 복합 원인
- 비관적 락이 근본 원인이라는 결론이 3회 연속 재확인됨
