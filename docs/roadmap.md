# Form-PASS 향후 개발 로드맵

## 1. 분산 락 도입 (Redis + Redisson)

### 배경

k6 스트레스 테스트(500 VU)에서 비관적 락 단일 행 경합으로 에러율 13.8%, p95 11.3초가 측정되었다.
커넥션 풀 튜닝(10→30)으로도 개선되지 않아, 병목 원인이 DB 레벨 락 자체임을 확인했다.

### 목표

- 락을 DB에서 Redis(애플리케이션 레벨)로 분리하여 DB 커넥션 점유 시간 최소화
- 동일 조건(500 VU) 재측정으로 Before/After 성능 비교

### 구현 방향

- Redis + Redisson 기반 분산 락 적용
- 트랜잭션 범위를 최소화하여 락 점유 시간 단축
- 단일 서버에서도 효과 검증 후, 스케일 아웃 대비 기반 마련

### 완료 기준

- 500 VU 스트레스 테스트에서 에러율 및 p95 응답시간 개선 수치 확보
- Before/After 비교 리포트 작성

## 2. 대기열 시스템 구축

### 배경

티켓팅 오픈 시 수백~수천 명이 동시 접속하면 서버에 요청이 한꺼번에 몰린다.
분산 락으로 데이터 정합성은 보장되지만, 사용자 경험 측면에서 대기 없이 타임아웃이 발생하거나 에러 페이지를 보게 된다.

### 목표

- 서버가 처리할 수 있는 만큼만 요청을 받고, 나머지는 대기열에서 순서대로 진입
- 사용자에게 현재 대기 순번과 예상 대기 시간을 제공

### 구현 방향

- Redis Sorted Set 기반 대기열 (진입 시각을 score로 사용)
- 폴링 또는 SSE로 대기 순번 실시간 전달
- 처리 가능 인원만큼 대기열에서 꺼내 예약 페이지 진입 허용

### 완료 기준

- 대기열 진입 → 순번 확인 → 예약 페이지 진입 흐름 구현
- 부하 테스트로 대기열 유무에 따른 에러율 비교

## 3. 서비스 품질 개선 (UX / 대시보드)

### 배경

현재 호스트 대시보드는 총 좌석 수, 예약 수, 체크인 수 등 기본 통계만 제공한다.
실제 운영 시 호스트가 필요로 하는 정보(시간대별 예약 추이, 노쇼율 등)가 부족하다.

### 목표

- 호스트가 이벤트 운영에 필요한 의사결정을 할 수 있는 수준의 정보 제공
- 게스트 예약 플로우의 UX 개선

### 개선 항목

**대시보드**
- 스케줄별 예약률 시각화 (어떤 시간대가 인기인지)
- 체크인율 / 노쇼율 통계
- 예약 추이 (일별/시간별)

**게스트 UX**
- 예약 완료 후 QR 코드 즉시 표시 개선
- 예약 확인 페이지 접근성 개선

### 완료 기준

- 호스트 대시보드 API 확장 및 프론트 반영
- 게스트 플로우 개선 사항 배포

## 4. 인프라 개선

### 배경

현재 t2.micro(vCPU 1, 1GB) 단일 인스턴스로 운영 중이다.
부하 테스트에서 500 VU 구간에서 TCP 연결 실패(dial: i/o timeout)가 발생하여 인스턴스 스펙 자체의 한계가 확인되었다.

### 개선 항목

- **프론트엔드 인프라 통합**: Vercel → S3 + CloudFront로 이전하여 백엔드와 동일 AWS 인프라로 통합, CORS 의존 최소화
- **SSL 인증서 관리 개선**: Nginx + Certbot → ALB + ACM 전환 검토 (인증서 자동 갱신, Nginx 관리 포인트 제거)
- **스케일 아웃 기반 마련**: 분산 락 + 대기열 완료 후, 다중 인스턴스 구성 검토

## 5. 문서 체계 구축 (PRD / ADR)

### 배경

지금까지 기능 구현과 기술적 의사결정이 코드와 커밋 로그에만 남아있다.
왜 비관적 락을 선택했는지, 왜 QueryDSL을 도입했는지 등의 판단 근거가 문서화되어 있지 않아
시간이 지나면 맥락이 사라진다.

### 목표

- 기능 요구사항은 **PRD(Product Requirements Document)**로 명세
- 기술적 의사결정은 **ADR(Architecture Decision Record)**로 기록
- 이후 개발 시 PRD → ADR → 구현 순서의 프로세스 정착

### 작성 대상

**PRD (docs/prd/)**
- 대기열 시스템 PRD
- 대시보드 개선 PRD
- 이후 신규 기능 개발 시 선행 작성

**ADR (docs/adr/)**
- ADR-001: 비관적 락 선택 배경과 한계
- ADR-002: QueryDSL 도입 배경
- ADR-003: 분산 락 도입 결정 (예정)
- 이후 아키텍처 변경 시 작성

### 완료 기준

- 기존 의사결정에 대한 ADR 소급 작성 (최소 2건)
- 다음 기능 개발부터 PRD 선행 작성 프로세스 적용

## 6. AI 기반 개발 프로세스 구축

### 배경

1인 개발 환경에서 AI 코딩 에이전트(Claude Code)를 활용하고 있으나,
명확한 규칙 없이 사용하면 코드 스타일 불일치, 불필요한 변경, 스펙 누락 등이 발생한다.
rule.md, CLAUDE.md를 통해 AI의 행동 규칙과 프로젝트 컨텍스트를 정의하여
일관된 품질의 코드를 생산하는 프로세스를 구축한다.

### 현재 구축 완료

- **rule.md**: AI 코딩 에이전트의 절대 규칙 (아키텍처, 코드 스타일, 테스트, 보안 등 13개 섹션)
- **CLAUDE.md**: 프로젝트 구조, 도메인 관계, 컨벤션, 빌드 명령어 등 프로젝트 컨텍스트
- **spec.md + docs/spec/**: 기능 명세 인덱스 및 상세 문서 (API, ERD, 비즈니스 규칙, 도메인)
- **개발 프로세스 정의**: 요구사항 정리 → 명세 문서화 → 구현 계획 → 승인 → 구현 → 테스트 → 커밋

### 추가 개선 방향

- PRD/ADR 문서 체계와 연동하여 AI가 기능 구현 전 관련 PRD/ADR을 참조하도록 프로세스 강화
- 부하 테스트 스크립트 작성, 리포트 생성 등 반복 작업의 AI 위임 범위 확대
- AI가 생성한 코드의 품질 검증 체크리스트 정비 (테스트 커버리지, 보안, 성능)

### 완료 기준

- PRD/ADR 작성 시 AI 에이전트가 참조해야 할 문서 경로와 규칙을 rule.md에 반영
- 신규 기능 개발 1건을 PRD → ADR → AI 구현 → 테스트 전체 프로세스로 완수하여 검증
